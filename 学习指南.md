# Demo 1 学习指南:一步步搭出工业级 CLI 基座

> 这份指南带你从零开始,每一步只引入一个新概念。
> **不直接给你完整代码**,而是告诉你"做什么、为什么、怎么验证"。
> 做完一步,验证通过,再进下一步。慢就是快。

---

## 总览

8 个步骤,每步 15-60 分钟,总计 5-6 小时:

| Step | 做什么 | 引入工具 | 跑通后能干嘛 |
|------|--------|---------|------------|
| 0 | 准备目录 | npm init | 有个空项目 |
| 1 | 写第一个 TS | tsc | 知道 TS 怎么编译成 JS |
| 2 | 直接跑 TS | tsx | 改完代码不用编译就能跑 |
| 3 | 做 CLI 命令 | commander | 能解析 `hello --name X` |
| 4 | 加构建打包 | tsup | 产出 dist/给别人用 |
| 5 | 写测试 | vitest | 改代码不怕改坏 |
| 6 | 加严格模式 | tsconfig 严格选项 | TS 帮你抓更多错 |
| 7 | 加覆盖率 | vitest coverage | 量化测试完整性 |
| 8 | (可选)CI + 发布 | changesets + GH Actions | 自动化发布 |

## Step 1: 写第一个 TS 文件,认识 tsc

### 🎯 目标

写一个 `.ts` 文件,用 `tsc` 编译成 `.js`,理解 TS → JS 的过程。

### 📝 操作

**1. 装 TypeScript**

```bash
pnpm add -D typescript
```

**2. 初始化 tsconfig**

```bash
npx tsc --init
```

会生成一个 `tsconfig.json`,内容很多(全是注释)。先不动。

**3. 写第一个 TS 文件**

创建 `src/hello.ts`:

```ts
function greet(name: string): string {
  return `Hello, ${name}!`;
}

console.log(greet("World"));
```

**4. 用 tsc 编译**

```bash
npx tsc --ignoreConfig src/hello.ts
```

> **TypeScript 6 新规则**:命令行指定文件时,需要 `--ignoreConfig` 明确告诉 tsc "忽略 tsconfig.json,用默认配置"。Step 0 生成的 tsconfig.json 在这一步只是占位,后面 Step 6 才正式配置它。

### ✅ 验证

```bash
ls src/
# 应该看到 hello.ts 和 hello.js

node src/hello.js
# 输出: Hello, World!
```

### 💡 解释

**发生了什么?**

```
hello.ts  ──[tsc]──▶  hello.js
   ↑                    ↑
TypeScript 源码       JavaScript(去掉类型)
```

- `tsc` 把 TS 翻译成 JS
- 它**不做类型检查**(默认),只是把 `: string` 这些类型标注擦掉
- 产物 `hello.js` 是普通 JS,Node 能直接跑

**为什么 `tsc` 只翻译不检查?** 因为没传 `--strict` 等选项。tsc 默认行为很宽松,后面 Step 6 会加严格模式。

### 🚧 常见错误

- `npx: command not found`:npx 是 npm 自带的,装 Node 就有。
- `error TS5055: Cannot write file 'src/hello.js' because it would be overwritten`:已经有 hello.js 了,先删 `rm src/hello.js` 再编译。

### 🤔 思考题(可选)

打开 `src/hello.js` 看看,`function greet(name: string)` 变成了什么?
答案:类型标注 `: string` 被擦除了,变成 `function greet(name)`。

## Step 2: 用 tsx 直接跑 TS

### 🎯 目标

改完代码不用每次编译,直接跑 TS。理解 `tsx` 解决了什么问题。

### 📝 操作

**1. 装 tsx**

```bash
pnpm add -D tsx
```

**2. 直接跑 TS**

```bash
npx tsx src/hello.ts
```

**3. 改代码再跑**

改 `src/hello.ts` 的 `"World"` 成 `"CLI"`,再跑一次:

```bash
npx tsx src/hello.ts
# 输出: Hello, CLI!
```

### ✅ 验证

改代码后**不用编译**,直接跑能输出最新结果。

### 💡 解释

**问题**:Step 1 每改一次代码都要 `tsc src/hello.ts` 编译,再 `node src/hello.js` 跑,太慢。

**tsx 解决了什么?**

```
hello.ts  ──[tsc]──▶  hello.js  ──[node]──▶  输出   (慢)
hello.ts  ──[tsx]──────────────────────────▶  输出   (快)
```

`tsx` 内部用 `esbuild`(极快的 TS 编译器),改完代码瞬间就能跑。

**类比**:你用 Vue/Vite 时,改代码自动热更新就是这种体验。tsx 是命令行版的。

### 🚧 常见错误

- `Error: Cannot find module 'tsx'`:没装,先 `pnpm add -D tsx`。

### 📝 加个 script

编辑 `package.json`,加 scripts:

```json
"scripts": {
  "dev": "tsx src/hello.ts"
}
```

然后可以 `pnpm dev` 代替 `npx tsx src/hello.ts`。

## Step 3: 加 commander,做成 CLI

### 🎯 目标

让代码能解析命令行参数,变成真正的 CLI 工具:
- `pnpm dev hello` → 输出 Hello, CLI!
- `pnpm dev hello --name World` → 输出 Hello, World!

### 📝 操作

**1. 装 commander**

```bash
pnpm add commander
```

**2. 改造代码**

把 `src/hello.ts` 改名成 `src/index.ts`(删除旧文件,新建),内容:

```ts
import { Command } from "commander";

const program = new Command();

program
  .command("hello")
  .option("--name <name>", "name to greet", "CLI")
  .action((opts: { name: string }) => {
    console.log(`Hello, ${opts.name}!`);
  });

program.parse();
```

**3. 修改 script**

```json
"scripts": {
  "dev": "tsx src/index.ts"
}
```

### ✅ 验证

```bash
pnpm dev hello
# 输出: Hello, CLI!

pnpm dev hello --name World
# 输出: Hello, World!

pnpm dev --help
# 输出: 帮助信息
```

### 💡 解释

**commander 是什么?**

CLI 工具的核心是"解析命令行参数"。`process.argv` 长这样:
```
['node', 'index.ts', 'hello', '--name', 'World']
```

自己解析太麻烦,commander 帮你做:
- 注册子命令(`.command("hello")`)
- 注册选项(`.option("--name <name>")`)
- 自动生成 `--help`
- 调用对应 action

**关键 API**:
- `new Command()` 创建一个命令对象
- `.command("hello")` 注册子命令 hello
- `.option("--name <name>", "描述", "默认值")` 注册选项
- `.action((opts) => {...})` 执行逻辑,opts 是解析后的参数
- `program.parse()` 触发解析(必须调用)

### 🚧 常见错误

- `Error: Cannot find module 'commander'`:没装,先 `pnpm add commander`。
- 报 `SyntaxError: Cannot use import statement`:package.json 没加 `"type": "module"`,回 Step 0 检查。

### 🤔 思考题

为什么 import 路径是 `"commander"` 不是 `"./commander"`?
答案:`commander` 是 npm 包,不带 `./` 或 `/` 开头的 import 都是去 `node_modules` 找。

## Step 4: 加 tsup 做构建

### 🎯 目标

把 TS 源码打包成 `dist/index.js`,让别人能直接 `node dist/index.js` 跑,不需要 tsx。理解为什么要构建。

### 📝 操作

**1. 装 tsup**

```bash
pnpm add -D tsup
```

**2. 创建 tsup.config.ts**

```ts
import { defineConfig } from "tsup";

export default defineConfig({
  entry: ["src/index.ts"],
  format: ["esm"],
  clean: true,
});
```

**3. 加 build script**

编辑 `package.json`,加 scripts:

```json
"scripts": {
  "dev": "tsx src/index.ts",
  "build": "tsup"
}
```

**4. 跑构建**

```bash
pnpm build
```

### ✅ 验证

```bash
ls dist/
# 应该看到 index.js

node dist/index.js hello --name Build
# 输出: Hello, Build!
```

### 💡 解释

**为什么需要构建?**

```
开发时:  src/*.ts  ──[tsx]──▶  跑起来  (你用,有 tsx)
发布后:  dist/*.js ──[node]──▶ 跑起来  (用户用,没 tsx)
```

用户装你的包,不会装你的 `src/`,只有 `dist/`。所以要把 TS 编译成 JS。

**tsup 是什么?**

打包工具,内部用 esbuild(极快的编译器),把:
- 多个 TS 文件 → 一个 JS 文件
- 擦除类型标注
- 生成 sourcemap(可选)

**配置解释**:
- `entry: ["src/index.ts"]`:入口文件
- `format: ["esm"]`:输出 ESM 格式(后面会加 cjs)
- `clean: true`:构建前清空 dist

### 🚧 常见错误

- `Error: Cannot find module 'tsup'`:没装,先 `pnpm add -D tsup`。
- `dist/index.js` 第一行没有 `#!/usr/bin/env node`:这没关系,等会儿讲 shebang。

### 📝 加 shebang

`dist/index.js` 第一行要有 `#!/usr/bin/env node`,这样能直接执行(`./dist/index.js hello`)。

做法:在 `src/index.ts` **第一行**加:

```ts
#!/usr/bin/env node
import { Command } from "commander";
// ... 其余代码
```

重新 `pnpm build`,检查 `dist/index.js` 第一行。

### 🤔 思考题

为什么开发用 tsx,发布用 tsup?
答案:tsx 不产出文件(只是运行时转译),tsup 产出 `dist/*.js` 文件给别人用。开发要快(tsx),发布要有产物(tsup)。

## Step 5: 写第一个测试

### 🎯 目标

给 `greet` 函数写单元测试,理解 `vitest` 怎么用。

### 📝 操作

**1. 重构:把逻辑抽出来**

目前逻辑写在 `action` 里,不好测。抽成纯函数。

创建 `src/greet.ts`:

```ts
export function buildGreeting(name: string): string {
  return `Hello, ${name}!`;
}
```

修改 `src/index.ts`:

```ts
#!/usr/bin/env node
import { Command } from "commander";
import { buildGreeting } from "./greet.js";

const program = new Command();

program
  .command("hello")
  .option("--name <name>", "name to greet", "CLI")
  .action((opts: { name: string }) => {
    console.log(buildGreeting(opts.name));
  });

program.parse();
```

**注意**:import 路径是 `"./greet.js"` 不是 `"./greet.ts"`!原因后面解释。

**2. 装 vitest**

```bash
pnpm add -D vitest
```

**3. 写测试**

创建 `tests/greet.test.ts`:

```ts
import { describe, it, expect } from "vitest";
import { buildGreeting } from "../src/greet.js";

describe("buildGreeting", () => {
  it("greets the given name", () => {
    expect(buildGreeting("World")).toBe("Hello, World!");
  });

  it("greets default name", () => {
    expect(buildGreeting("CLI")).toBe("Hello, CLI!");
  });
});
```

**4. 加 test script**

```json
"scripts": {
  "dev": "tsx src/index.ts",
  "build": "tsup",
  "test": "vitest run"
}
```

**5. 跑测试**

```bash
pnpm test
```

### ✅ 验证

看到:
```
Test Files  1 passed (1)
     Tests  2 passed (2)
```

### 💡 解释

**为什么要抽成纯函数?**

```
不好测:  action(opts) → console.log(结果)    (要 mock console)
好测:    buildGreeting(name) → 返回字符串     (直接断言返回值)
```

把逻辑从 IO(console.log)里抽出来,变成纯函数,测试就简单了。

**为什么 import 路径是 `.js` 不是 `.ts`?**

TS 编译后 `.ts` → `.js`,import 路径不变。如果写 `"./greet.ts"`,编译后还是 `"./greet.ts"`,但实际文件已经是 `.js` 了,找不到。

ESM 规范要求 import 必须带后缀。写 `.js` 是约定(源文件 `.ts`,但 import 写编译后的后缀)。tsup 和 vitest 都能正确解析。

**vitest API**:
- `describe("组名", () => {...})`:测试分组
- `it("用例名", () => {...})`:单个测试用例
- `expect(值).toBe(期望)`:断言

### 🚧 常见错误

- `ERR_MODULE_NOT_FOUND: Cannot find module '../src/greet.js'`:你 import 写成了 `.ts`,改成 `.js`。
- `Cannot find module 'vitest'`:没装,先 `pnpm add -D vitest`。

### 🤔 思考题

为什么测试文件叫 `greet.test.ts` 不叫 `greet.ts`?
答案:vitest 默认只跑 `.test.ts` 或 `.spec.ts` 文件,避免误跑普通文件。

## Step 6: 加 TypeScript 严格模式

### 🎯 目标

打开 TS 的严格选项,让类型检查帮你抓更多错。理解 `strict: true` 是什么。

### 📝 操作

**1. 编辑 tsconfig.json**

把 `tsconfig.json` 改成(删掉所有注释,只留这些):

```json
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "ESNext",
    "moduleResolution": "Bundler",
    "strict": true,
    "esModuleInterop": true,
    "skipLibCheck": true,
    "noEmit": true
  },
  "include": ["src/**/*.ts", "tests/**/*.ts"]
}
```

**2. 加 typecheck script**

```json
"scripts": {
  "dev": "tsx src/index.ts",
  "build": "tsup",
  "test": "vitest run",
  "typecheck": "tsc --noEmit"
}
```

**3. 跑类型检查**

```bash
pnpm typecheck
```

### ✅ 验证

没有输出(或只输出 `>` 提示符),说明类型检查通过。

### 💡 解释

**`tsconfig.json` 是什么?**

TypeScript 的配置文件,告诉 `tsc`:
- 检查哪些文件(`include`)
- 检查多严(`strict`)
- 怎么编译(`target`, `module`)

**关键选项**:
- `strict: true`:打开一组严格检查(下面细说)
- `noEmit: true`:不输出 JS 文件,只做类型检查
  - 为什么?因为编译交给 tsup 了,tsc 只负责检查
- `moduleResolution: "Bundler"`:用 bundler 的方式解析 import(允许省略后缀)
- `skipLibCheck: true`:跳过 `.d.ts` 检查,提速

**`strict: true` 等价于开了 7 个选项**:
1. `noImplicitAny`:禁止隐式 any,参数必须标类型
2. `strictNullChecks`:`null`/`undefined` 不再是所有类型的子类型
3. `strictFunctionTypes`:函数参数严格检查
4. 其他 4 个...

**`strictNullChecks` 最重要**:

```ts
// 不开 strictNullChecks:
const arr = [1, 2, 3];
const x = arr.find(n => n === 5);  // 类型:number(实际可能是 undefined)
x + 1;  // 不报错,但运行时崩

// 开了 strictNullChecks:
const x = arr.find(n => n === 5);  // 类型:number | undefined
x + 1;  // 报错!必须先检查 x !== undefined
```

### 🚧 常见错误

- 报一堆类型错误:正常,`strict` 让以前不报的错现在报了。一个个修。
- `error TS18003: No inputs were found`:tsconfig 的 `include` 没匹配到文件,检查路径。

### 🤔 试一下

故意改错类型,看 tsc 怎么报:

```ts
// src/greet.ts
export function buildGreeting(name: string): string {
  return 123;  // 故意返回数字
}
```

跑 `pnpm typecheck`,应该报错。改回来。

## Step 7: 加覆盖率

### 🎯 目标

看测试覆盖了多少代码,理解覆盖率是什么、为什么卡阈值。

### 📝 操作

**1. 装覆盖率工具**

```bash
pnpm add -D @vitest/coverage-v8
```

**2. 创建 vitest.config.ts**

```ts
import { defineConfig } from "vitest/config";

export default defineConfig({
  test: {
    coverage: {
      provider: "v8",
      reporter: ["text"],
      include: ["src/**/*.ts"],
    },
  },
});
```

**3. 加 coverage script**

```json
"scripts": {
  "dev": "tsx src/index.ts",
  "build": "tsup",
  "test": "vitest run",
  "test:coverage": "vitest run --coverage",
  "typecheck": "tsc --noEmit"
}
```

**4. 跑覆盖率**

```bash
pnpm test:coverage
```

### ✅ 验证

看到类似输出:
```
File      | % Stmts | % Branch | % Funcs | % Lines |
----------|---------|----------|---------|---------|
All files |   100%  |   100%   |  100%   |  100%   |
 greet.ts |   100%  |   100%   |  100%   |  100%   |
```

### 💡 解释

**覆盖率是什么?**

测试跑过的代码占总代码的比例。四种指标:
- **Lines**:多少行被执行了
- **Statements**:多少语句执行了
- **Functions**:多少函数被调用了
- **Branches**:多少 if/else 分支被走了

**为什么要有覆盖率?**

```
没覆盖率:  你以为测了,实际漏了某个 if 分支
有覆盖率:  数据告诉你哪行没测到
```

**先不卡阈值**:Step 7 只是"看"覆盖率。后面 Step 8(可选)再加阈值,逼自己写够测试。

### 🚧 常见错误

- `Cannot find module '@vitest/coverage-v8'`:没装,先 `pnpm add -D @vitest/coverage-v8`。

### 🤔 试一下

故意删一个测试用例,看覆盖率掉到多少:

```ts
// tests/greet.test.ts 删掉第二个 it
describe("buildGreeting", () => {
  it("greets the given name", () => {
    expect(buildGreeting("World")).toBe("Hello, World!");
  });
  // 删掉这个:
  // it("greets default name", () => {...});
});
```

跑 `pnpm test:coverage`,看覆盖率还是不是 100%?(答案:还是,因为两个用例测的是同一行代码,删一个不影响)

这说明**100% 覆盖率不等于测全了**。覆盖率只是参考,不是真理。

## Step 8(可选):加 CI 和发布

> 这一步可选。如果你还没准备发布到 npm,可以跳过,直接进 Demo 2。

### 🎯 目标

- 用 changesets 管理版本
- 配 GitHub Actions 自动跑 CI
- 发布到 npm

### 📝 操作

**1. 装 changesets**

```bash
pnpm add -D @changesets/cli
pnpm changeset init
```

**2. 创建 .github/workflows/ci.yml**

```yaml
name: CI
on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  ci:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4
        with:
          version: 10
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: pnpm
      - run: pnpm install --frozen-lockfile
      - run: pnpm typecheck
      - run: pnpm test
      - run: pnpm build
```

**3. 创建 GitHub 仓库并推送**

```bash
cd /Users/doce/project/terminal
git init
git add 01-cli-starter/
git commit -m "feat: init cli-starter"
# 在 github.com 创建仓库后
git remote add origin <your-repo-url>
git push -u origin main
```

**4. 走一次 changeset 流程**

```bash
cd 01-cli-starter
pnpm changeset       # 交互式描述变更
git add .changeset/
git commit -m "chore: add changeset"
pnpm version         # 消费 changeset,升版本
pnpm release         # build + publish(需要 npm login)
```

### ✅ 验证

- GitHub 仓库的 Actions 标签页能看到 CI 跑通(绿勾)
- `npx @fe-copilot/cli-starter hello` 能跑(发布后)

### 💡 解释

**changesets 是什么?**

版本管理工具。流程:
1. 改代码
2. `pnpm changeset` 描述"改了什么"(patch/minor/major)
3. 生成一个 markdown 文件,提交到 git
4. 累积几个 changeset 后,`pnpm version` 自动升版本号 + 生成 CHANGELOG
5. `pnpm release` 发布到 npm

**为什么用 changesets 而不是手动改版本号?**
- 自动生成 CHANGELOG
- 支持 monorepo(后面 fe-copilot-cli 就是)
- 不会忘改版本号

**CI 干什么?**
- 每次 push / PR 自动跑 typecheck + test + build
- 任一步失败 CI 红,不能合并
- 保证主分支永远可发布

### 🚧 常见错误

- CI 报 `pnpm: command not found`:action-setup 顺序错了,必须在 setup-node 之前。
- `npm publish` 报 403:没登录,先 `npm login`。
- provenance 错误:本地 publish 不支持 provenance,要么去掉这个配置,要么从 CI 发布。

## 学习原则回顾

1. **一步一验证**:做完一步,验证通过,再进下一步
2. **卡壳了别跳过**:回头看那一步的"常见错误",还是不懂问我
3. **故意改错试试**:每步都有"思考题"或"试一下",做了才真理解
4. **不追求一次配齐**:Step 6 之前 tsconfig 都是默认宽松的,跑通了再加严格
